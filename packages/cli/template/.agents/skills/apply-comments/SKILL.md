# apply-comments

Use when the user has interacted with the in-browser Inspector (`Cmd/Ctrl+I` → click → leave a note) and the dev server has injected `// @canvas-comment[id=...]: ...` markers into `boards/**/*.tsx`. The job is to **turn each marker into a real code change and delete the marker**, closing the feedback loop.

Read `board-authoring` before editing; this file does not restate its rules.

## What a marker looks like

The Inspector injects each comment as a single-line C-style comment **directly above** the JSX element it targets:

```tsx
  // @canvas-comment[id=cmt_01HXY]: make the heading red
  <h1 className="text-7xl font-bold">Hello Canvas</h1>
```

The `id` is generated by the dev server and is unique per comment. Multiple markers may stack above the same element when the user leaves a sequence of notes.

## Process

1. **Find every marker**: `Grep` `boards/**/*.tsx` for `@canvas-comment` and list `[file, line, id, comment text]`.
2. **Process in source order** (do not reorder — stacked markers on the same element form a temporal sequence; out-of-order processing produces contradictions):
   1. Read 3 lines above the marker and 10 lines below for context.
   2. Decide what the comment refers to and what to change. **If you cannot tell, skip the marker** — leave it in place and add it to a "skipped" list for the user. Do not guess.
   3. Make the **smallest possible** change to the JSX element on the next line. Match the comment's intent; do not piggyback unrelated cleanups.
   4. **Delete the entire `// @canvas-comment[id=...]: ...` line.** A surviving marker means the comment was never resolved (`AGENTS.md` invariant §5).
3. **One commit per id** when commits are appropriate, with message `apply <id>: <one-line summary>`.
4. **Report** to the user: N applied, M skipped (with reasons for each skip).

## Hard rules

- **The marker must be deleted** when the change is done. Leaving it equals a no-op from the user's perspective — the same feedback will reappear next time they look.
- **No batched refactors.** "While I'm here, let me clean up this block" is a trap — comments can contradict each other; resolve one at a time.
- **Do not rename ids or move sections.** Inspector's `data-oc-loc` reverse-lookup uses file/line/column; relocating code invalidates every unprocessed marker (`AGENTS.md` invariant §1).
- **On conflict, apply and warn once.** If a comment asks for something that violates `board-authoring` (e.g. "remove all padding"), do what the user asked, but tell them once that it conflicts with the rule — they can decide.

## Legitimate reasons to skip

- The comment text is ambiguous ("this feels off" — off how?).
- The target element no longer exists (the user may have edited code after leaving the comment).
- The change requires a new dependency or edits outside the board file — out of scope; let the user decide.

Skipping is not failure; it is responsible. Just list the skipped ids clearly.
